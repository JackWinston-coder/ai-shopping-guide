# 项目执行规则

## 目的

本规则用于约束项目推进节奏，确保开发过程严格遵循已确认的开发计划，不随意跳阶段、不扩大范围、不提前引入未确认内容。

## 执行原则

1. 只允许按当前 Milestone 和当天小目标推进。
2. 每次只做一个小步骤，不同时展开多个大任务。
3. 不得擅自跳到后续阶段，例如在 UI 原型未完成前直接进入 RAG、爬虫、部署或复杂后端联通。
4. 任何涉及偏离计划、改技术路线、加新功能、换实现方式的决定，必须先说明原因并征求确认。
5. 每次开始前先说明下一步要做什么，确认后再执行。
6. 每次完成后更新 `devlogs/`，记录已完成事项、问题、风险、待办。

## 允许的推进方式

- 当前阶段只做当前阶段内容。
- 当前阶段任务只拆成可在一次工作中完成的小步骤。
- 如果发现更好的方向，可以提出建议，但不能直接切换执行。
- 如果发现原计划不合理，先暂停并提出变更建议。

## 偏离处理

当出现以下情况时，必须停止继续推进并先确认：

- 要新增未写入计划的功能。
- 要改变技术栈或架构。
- 要跨阶段执行任务。
- 要把“计划”变成“实现”但用户未确认。
- 要修改既有目标优先级。

## 推荐提示语

以后用户只要说以下任一表达，就默认按本规则执行：

- “按开发计划严格推进”
- “不要跳阶段”
- “先做当前 Milestone”
- “按执行规则来”

## 当前默认约束

- 当前只推进已确认的开发计划。
- 当前优先级高于未来阶段的一切新想法。
- 项目推进以稳定、安全、可验证为第一目标。
